Modelled on how Team Cherry (Hollow Knight, 2 devs + 1 artist) and Maddy Thorson's Celeste team (4 people) divided work. Each person owns a domain — you are never waiting for permission to make a decision inside your domain.

You — Product Lead + Lead Dev
Owns: final product decisions, Unity architecture, all game systems (player controller, enemy AI, combat), GitHub repo management, sprint planning, milestone sign-off. You are the single source of truth on "does this go in the game." Every disagreement ends with your call — no votes, no committees. Inspired by Toby Fox (Undertale) who was sole decision-maker even with collaborators.

Phase 0 — Pre-production (Weeks 1–4)
Exit condition: every team member can open the Miro board and answer "what am I building this week" without asking you. This is what Eric Barone (ConcernedApe) said he wished he'd done before starting Stardew Valley's code — a full documented design before touching the engine.
You: Finalise game design document. Write the core mechanics spec (player controller rules, combat numbers, camera rules). Set up GitHub repo and Unity project skeleton. Define folder structure (see Unity tab).

D1: Deliver style bible — 1 hero character sheet for Amari, colour palette (max 5 colours + shades), line weight rule, and 3 enemy silhouettes.

D2: Deliver 2 level layout mockups (Miro) for levels 1 and 4. No art yet — boxes and labels only.

D3: Set up Miro board with all 5 zones. Deliver UI wireframes for main menu, HUD, and game-over screen. Source 10 placeholder sound FX from freesound.org.

Dev 2: Set up Notion workspace. Create sprint template. Research Unity packages needed (Cinemachine, TextMeshPro, Input System).
Phase 1 — Prototype (Weeks 5–10)
Exit condition: a playable grey-box level with all core mechanics functional. No art. Any team member can play it and understand the game loop. Hollow Knight's Team Cherry spent 6 months in pure prototype before touching final art — this discipline is why the game feels tight.

You: Build player controller (movement, jump, wall-jump, dash). Build basic enemy AI (patrol, detect, attack). Build combat system (melee, spear throw). Camera follow with Cinemachine.

Dev 2: Build main menu scene, health UI, score display. Wire placeholder audio. Set up scene loader.

Designers: Produce final art for Level 1 only. D1 animates Amari (idle, run, jump, attack — minimum viable set). D2 delivers Level 1 tileset. D3 delivers HUD visual design and first 5 real sound FX.

End of Phase 1 deliverable: a fully playable Level 1 with real art. This becomes your demo for any future funding or publisher conversations.
Phase 2 — Alpha (Weeks 11–30)
Exit condition: all 20 levels exist in grey-box form, 5 levels have final art, and the core mechanical loop has been playtested by at least 10 people outside the team.

This is your longest phase — protect it from feature creep. Celeste's team used a strict rule: no new mechanics after week 8 of alpha. Any new idea goes on a "maybe list" — reviewed only at phase end.

Cadence: 2-week sprints. Each sprint ships at least 2 new grey-box levels and progresses 1 level to final art. You hold a 30-minute sprint review every second Friday. No exceptions — the rhythm creates accountability without management overhead.
Phase 3 — Beta (Weeks 31–40)
Exit condition: all 20 levels have final art and are bug-free on the target platform. A build has been sent to 20 external playtesters and their feedback has been actioned.

Feature lock at week 31 — nothing new enters the game. The only allowed work is: bug fixing, polish (juice, particle FX, screen shake), performance optimisation, and sound mix. This is the phase most indie teams skip and then regret. Undertale's 1.0 launch was noticeably rough; Toby Fox spent months on patches that a proper beta would have caught.
Phase 4 — Launch (Weeks 41–44)
Exit condition: submitted to target stores, trailer published, press kit live.

You: Final builds for all platforms. Submission checklist (Google Play / App Store / itch.io / Steam). Store page copy and screenshots.

D3: Trailer editing, press kit design (logo, screenshots, one-sheet). Social media asset pack.

Dev 2: Analytics integration (if applicable). Crash reporting setup. Launch day monitoring plan.

Git workflow — 3-branch rule
main — always a playable build. Nobody commits directly here.

dev — integration branch. You and Dev 2 merge feature branches here. Builds cut from dev every Friday.

feature/[name] — one branch per feature. Examples: feature/shadow-mechanic, feature/level-03. Never merge to main directly — always through dev with your review.
ScriptableObjects — your secret weapon
Use ScriptableObjects for all game data that designers might want to tweak: enemy health, weapon damage, spawn rates, ability cooldowns. This lets Dev 2 or even designers adjust balance without touching code. Hollow Knight used this pattern extensively — non-programmers could tune combat feel without you.

Create a EnemyStatsSO.cs for every enemy type. Store instances in ScriptableObjects/EnemyStats/.
Unity packages to install at project start
Cinemachine — camera follow, screen shake, zoom. Replaces all manual camera code.
Input System (new) — cross-platform input. Set up once, works on PC, Android, iOS without code changes.
TextMeshPro — all in-game text. Crisp at any resolution; standard fonts look bad on mobile.
2D Animation (Sprite Rigging) — bone-based animation for Amari. Reduces the number of animation frames designers need to draw.
Universal Render Pipeline (URP) — better 2D lighting. Use this from day one — switching pipelines mid-project is a nightmare (ask any Unity dev who's done it).
Unity Analytics or GameAnalytics SDK — install early even if you don't use it yet. Retroactively adding analytics to an existing codebase is painful.
Tool stack — one tool per job, no overlaps
The biggest indie team killer is using 4 tools for the same job. Each tool below is the only tool for its category. If a new tool is proposed, it replaces an existing one — it never joins it.
Unity (2022 LTS) — game engine
Use the Long Term Support version, not the latest. Stability over features. Lock the Unity version in the project on day one — everyone must use the same version or you'll fight phantom bugs for months.
Why: 2D tooling, huge asset store, dominant mobile export. Used by Hollow Knight, Cuphead, Ori.
GitHub + Git LFS — version control
Git LFS (Large File Storage) is mandatory for Unity — without it, binary assets bloat the repo and break everything. Set up .gitattributes for all art formats on day one: .png, .psd, .wav, .mp3, .anim, .unity, .prefab.
Why: free for small teams, GitHub Desktop UI makes it accessible to designers who haven't used Git before.
Notion — task management + documentation
Two databases: Sprint Backlog (Kanban view with columns: Backlog / This Sprint / In Progress / In Review / Done) and Design Decisions Log (every major decision recorded with the date, who made it, and why — this prevents "I thought we decided X" arguments at month 3).
Why: free tier covers your team size. More flexible than Trello, less heavy than Jira. Toby Fox used Google Docs — Notion is the modern equivalent with database features.
Miro — visual planning + design review
Already in your workflow. Keep it as the visual layer only — level layout mockups, art style boards, sprint retrospective notes, the milestone timeline. Do not use it for task tracking (that's Notion's job).
Why: your designers live in visual tools. Miro meets them where they work instead of forcing them into Notion's text-heavy interface.
Aseprite — pixel art and animation
If your art style is pixel art (recommended for a 5-person first-time team — scope control). Aseprite exports sprite sheets directly in formats Unity expects. D1 and D2 work here; output goes to the shared drive /READY-FOR-DEV folder.
Why: $19.99 one-time, purpose-built for game sprites, directly imports into Unity via sprite sheet slicing. Stardew Valley was built entirely in Aseprite.
FMOD or Unity's built-in Audio Mixer — sound
Start with Unity's built-in Audio Mixer — it's simpler and more than enough for your scope. Only move to FMOD if you need adaptive music (music that changes based on game state, like Celeste's dynamic layers). D3 exports audio as .wav files; Dev 2 imports and wires in Unity.
Why: avoid introducing FMOD complexity until you need it. Most first-time teams over-engineer audio.
WhatsApp / Discord — async communication
One channel only for each purpose. #general (social), #dev-log (code updates), #art-wip (design work in progress, no feedback allowed here — feedback only in sprint review), #blockers (if something is stuck, post here and anyone can help). Disable notifications outside 8am–8pm — protect deep work time.
Why: communication overhead kills small teams. Structured channels prevent decision fatigue.
Real-world indie references — lessons for your team
Every working practice in this guide is drawn from documented accounts of how successful indie studios operated. These are the primary case studies.
Hollow Knight — Team Cherry
Ari Gibson (art), William Pellen (code), Jack Kelly (marketing) — 3 people, part-time nights and weekends, 2.5 years
Key lesson for you: They kept a strict "one mechanic at a time" rule. A feature wasn't added to the sprint until the previous one was fully integrated and playtested. This is why the game feels mechanically clean. Apply this to Freedom's Wrath: finish Shadow Mechanic before starting Spirit Sight, even if both are designed.
Celeste — Maddy Thorson + Noel Berry
2 developers + composer Lena Raine + artist Pedro Medeiros — built in 4 days as prototype, 7 months to ship
Key lesson for you: Their "playable Friday" rule is the most important thing you can steal. Every Friday the game must be playable. Not "mostly done" — playable. This catches integration bugs, keeps morale high (you see progress weekly), and forces honest scope cuts before they become crises.
Stardew Valley — ConcernedApe (Eric Barone)
1 developer, 4.5 years solo, outsourced music at the end
Key lesson for you: He built the entire game before releasing it — no early access, no crowdfunding. But more importantly, he maintained a strict daily development log. Every day he wrote what he did, what he planned tomorrow, and what was blocked. You should do this as a shared Notion doc your team can read. It creates accountability without micromanagement, and is a goldmine for postmortem reflection.
Shovel Knight — Yacht Club Games
Originally 4 ex-WayForward developers, grew to 12 — started with a Kickstarter
Key lesson for you: Their postmortem is famous for one line — "We cut a third of our planned content and the game is better for it." They had planned 12 enemy types per world and shipped 6. For Freedom's Wrath: your 20-level structure is ambitious for 2 devs. Define your minimum viable version (10 levels, 2 acts) before you start. That way, if life happens, you ship something — not nothing.
Undertale — Toby Fox
Primarily 1 developer, art outsourced to Temmie Chang, Kickstarter funded
Key lesson for you: Toby made one rule — write the story completely before building the game. He had the ending of Undertale written before level 1 was playable. For Freedom's Wrath, you've already done this (your 20-level narrative is fully designed). The lesson is: protect the story. When a mechanic feels fun but doesn't serve the story, cut the mechanic — not the story.
A Short Hike — Adam Robinson-Yu
1 developer, 4 months, intentionally small scope
Key lesson for you: He documented his development publicly on Twitter throughout — no publisher, no marketing budget, just showing work. By launch he had an audience. You are building a story that is culturally specific and meaningful. Document it publicly on social media from day one. Kenyan game dev is an underrepresented story that people will follow.
Non-negotiables — rules that protect the project
These are the decisions that must be made once and never revisited. Successful indie teams protect their energy by making these rules before problems arise, not after.
1
Never extend a sprint. If work doesn't fit, cut scope. If you extend sprints, your milestones become fiction and the team stops trusting deadlines. Celeste's team cut 3 full levels from the shipped game — the game is better for it.
2
Feature lock at alpha. Once Phase 2 begins, no new mechanics. New ideas go on a "next game" list. Freedom's Wrath has 20 levels and 8+ mechanics — it does not need more ideas. It needs execution.
3
One source of truth for every type of information. Assets live in Google Drive. Tasks live in Notion. Visual plans live in Miro. Code lives in GitHub. Nothing lives in anyone's personal laptop. If it's not in the shared system, it doesn't exist.
4
Playtest with strangers every month. Your team will go blind to your own game within weeks. Find 3 people outside the team every month and watch them play silently — don't explain anything, don't help them. What they get stuck on is your UI/UX problem, not their problem.
5
You are the final decision-maker on the product. Designers own the aesthetics within the style bible. Never design by committee — it produces average results. The Lead Artist's visual decisions are final in her domain. Your game design decisions are final in yours. Overlaps are resolved by you, once, in writing.
6
The main branch always builds. A broken main branch is a team emergency — drop everything, fix it immediately. This is a rule at every professional studio from Rockstar to indie teams of 3. A broken build destroys morale and wastes everyone's time.
7
Document decisions, not opinions. The Notion Design Decisions Log records what was decided and why — not who wanted what. This prevents the "but I thought we agreed" argument that derails teams at month 4. Every major game design choice gets a 3-line entry: decision, reason, who made the call.
8
Ship a minimum viable version first. Define 10 levels as your floor before starting. If life, money, or time forces a change — you ship 10 great levels rather than 20 unfinished ones. You can always add more post-launch (Hollow Knight shipped 4 free DLC expansions after launch).
